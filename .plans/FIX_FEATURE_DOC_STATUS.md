# Fix: mapping.md marks every feature "undocumented"

## Symptom

After running `find-the-gaps analyze`, the generated report always claims
every feature is undocumented:

- `mapping.md` renders `**Documentation status:** undocumented` for every
  feature.
- `docsfeaturemap.json` often records empty `pages: []` for every feature.

This happens even when the docs clearly cover the feature.

## Diagnosis

There are **two feature-naming universes** in the pipeline:

1. **Canonical code features** — produced by `ExtractFeaturesFromCode`
   (`internal/analyzer/code_features.go`). These names are the source of
   truth and are what appear in `FeatureMap.Feature.Name`,
   `featuremap.json`, and the headings in `mapping.md`.
2. **Raw per-page feature strings** — produced independently by
   `AnalyzePage` (`internal/analyzer/analyze_page.go`) and stored on each
   `PageAnalysis.Features`. These are free-form noun phrases the LLM
   extracts page-by-page, with no reference to the canonical list.

### Primary bug: reporter compares across universes

`reporter.WriteMapping` (`internal/reporter/reporter.go:34-45`) determines
doc status with a strict string equality check:

```go
for _, p := range pages {
    for _, f := range p.Features {
        if f == entry.Feature.Name { // canonical name == raw page feature
            docStatus = "documented"
            docPages = append(docPages, p.URL)
        }
    }
}
```

`entry.Feature.Name` is a canonical-universe string. `p.Features[i]` is a
raw-universe string. These are generated by two unrelated LLM calls. They
almost never collide exactly, so `docStatus` stays `"undocumented"` for
every feature.

### Correct source already exists but is ignored

`MapFeaturesToDocs` (`internal/analyzer/docs_mapper.go`) already does the
right thing: it feeds the canonical feature list into each per-page prompt
and asks the LLM for **exact matches from the list above**. The output,
`DocsFeatureMap`, is keyed on canonical feature names and contains the
pages that cover each.

`cli/analyze.go` builds this `docsFeatureMap`, caches it as
`docsfeaturemap.json`, and threads it into drift detection and
`WriteGaps` via `docCoveredFeatures`. But `WriteMapping` is still called
with `analyses []PageAnalysis` instead — so the reporter re-derives doc
status from the wrong source and gets it wrong.

### Secondary concern: `docsfeaturemap.json` exact-match fragility

`mapPageToFeatures` asks the LLM to return "exact matches from the list
above". Even with the canonical list in context, LLMs sometimes drift on
punctuation, casing, or pluralization. The drain loop in
`MapFeaturesToDocs` silently drops anything that isn't a verbatim key
(`if _, known := acc[feat]; known`), so near-misses disappear without a
trace. Result: `docsfeaturemap.json` entries can be empty even when the
page actually covers the feature.

This is a smaller, separate failure mode, but it contributes to the same
user-visible symptom once the primary bug is fixed.

## Plan

Follow the TDD rules in `CLAUDE.md`. Every step begins with a RED test.

### Step 1 — RED: Reporter test for doc status driven by `DocsFeatureMap`

1. Add a failing test in `internal/reporter/reporter_test.go` that:
   - Builds a `FeatureMap` with feature `"gap analysis"`.
   - Builds a `DocsFeatureMap` with entry
     `{Feature: "gap analysis", Pages: ["https://docs.example.com/gap"]}`.
   - Calls the new `WriteMapping` signature (see Step 2).
   - Asserts `mapping.md` contains `**Documentation status:** documented`
     and the page URL.
2. Add a second failing test that passes a `DocsFeatureMap` whose entry
   for the feature has `Pages: []`, asserts `undocumented` and `_(none)_`.
3. Modify the existing `TestWriteMapping_RichFields_Documented` /
   `_Undocumented` tests to use the `DocsFeatureMap` parameter instead of
   `PageAnalysis`. The old `PageAnalysis` path will be removed — these
   tests must fail first, then pass after Step 2.
4. Run `go test ./internal/reporter/...`. Confirm RED for the right
   reason (compile error or wrong doc status).

### Step 2 — GREEN: Switch `WriteMapping` to consume `DocsFeatureMap`

1. Change the signature:

   ```go
   func WriteMapping(
       dir string,
       summary analyzer.ProductSummary,
       mapping analyzer.FeatureMap,
       docsMap analyzer.DocsFeatureMap,
   ) error
   ```

   Drop the `pages []analyzer.PageAnalysis` parameter entirely.
2. Build `map[string][]string` (feature → pages) from `docsMap` once at
   the top of the function.
3. For each `entry`, look up `entry.Feature.Name` in that map. If the
   lookup returns a non-empty slice, set `docStatus = "documented"` and
   list the pages. Otherwise `undocumented` / `_(none)_`.
4. Update the call site in `internal/cli/analyze.go:333` to pass
   `docsFeatureMap` instead of `analyses`.
5. Run `go test ./internal/reporter/... ./internal/cli/...`. All should
   go GREEN.

### Step 3 — RED: Integration test covering the cross-universe mismatch

1. Add a test in `internal/reporter/reporter_test.go` (or a new
   `internal/cli/analyze_report_test.go`) that simulates the real
   failure mode: canonical feature name `"gap analysis"`, a
   `DocsFeatureMap` entry mapping that canonical name to a page, and
   `PageAnalysis.Features` containing an unrelated raw phrase like
   `"finding documentation gaps"`. Assert the feature is marked
   `documented`.
2. Confirm the test passes after Step 2 (it should — this just locks in
   the contract).

### Step 4 — Harden `MapFeaturesToDocs` against near-misses

Only do this if Step 2 fixes `mapping.md` but `docsfeaturemap.json` is
still empty after a real run against a fixture (Scenario 1 or 8 from
`.plans/VERIFICATION_PLAN.md`). Gate on evidence, not speculation.

1. RED: Test in `internal/analyzer/docs_mapper_test.go` where a fake
   `LLMClient` returns `["Gap Analysis"]` (different case) while the
   canonical feature is `"gap analysis"`. Assert the page is matched.
2. GREEN: In the drain loop, normalize both the canonical keys and each
   returned feature string with a small helper
   (`strings.ToLower` + `strings.TrimSpace` + collapse internal
   whitespace) and match through the normalized form while preserving
   the canonical key as the map key.
3. Log at `Warn` level any returned feature the LLM produced that still
   doesn't match any canonical key, so future misses are visible.
4. Re-run `go test ./...`.

### Step 5 — Verification

Run the real-world verification scenarios against a fixture with a known
docs page covering a known feature:

- `.plans/VERIFICATION_PLAN.md` Scenario 1 (happy path): after running
  `find-the-gaps analyze`, open `mapping.md` and confirm at least one
  feature shows `Documentation status: documented` with a real URL.
- Scenario 8 (real docs site): same check on real data.

If every feature is still `undocumented` after Step 2, stop and
re-investigate — the primary fix did not land.

  fix is in how `WriteMapping` consumes existing data.
- No change to drift detection — it already consumes
  `docCoveredFeatures` derived from `docsFeatureMap`.
