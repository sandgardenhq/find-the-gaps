# Verification Plan

This plan validates that **Find the Gaps** works against real codebases, a real docs site, and the real `mdfetch` binary. No mocks. No fakes. A test environment is acceptable only if it's a fully running copy of the system being integrated with.

## Prerequisites

Before any scenario runs:

- **Binaries on `$PATH`**
  - `find-the-gaps` (built from this repo — `go build ./cmd/find-the-gaps`)
  - `mdfetch` — `mdfetch --version` must succeed
- **Bifrost credentials** — a valid Bifrost API key exported as the env var the SDK expects. Verification will make real LLM calls; do not stub.
- **Sample fixture repo** — a checked-out copy of a small, real-world Go library with a README and a `docs/` folder. Suggested: a frozen clone of an open-source project pinned to a known commit, stored under `testdata/fixtures/` or generated by a script.
- **Sample docs site** — a public documentation URL reachable from the machine running verification. Suggested: the upstream project for the fixture repo.
- **Network** — the machine must be able to reach the docs URL and the Bifrost endpoint.
- **Clean state** — before first-run scenarios, remove `~/.find-the-gaps/` (or the project-local `.find-the-gaps/`) so the first-run banner fires.

## Scenarios

### Scenario 1: Happy Path — No Gaps

**Context**: Fixture repo and docs site are in known-good sync (pinned commit + matching docs snapshot).

**Steps**:
1. Run `find-the-gaps analyze --repo ./testdata/fixtures/known-good --docs-url https://<known-good-docs>`.
2. Let the tool run to completion.
3. Inspect the generated report.

**Success Criteria**:
- [ ] Exit code is `0`.
- [ ] Report produces zero findings, or only findings explicitly marked as acceptable (e.g., empty list, "no gaps detected" summary).
- [ ] Total runtime is bounded (target: under 2 minutes for the small fixture).

**If Blocked**: If findings are non-empty and unexpected, stop — this is a false-positive signal that invalidates every other scenario. Ask the developer to review the fixture.

---

### Scenario 2: Detect Undocumented Public Surface

**Context**: Start from the known-good fixture (Scenario 1). Add a new exported Go function with a doc comment but no corresponding entry in the docs site.

**Steps**:
1. In the fixture repo, add a new exported function (e.g., `func Frobnicate(x string) error { ... }`) in a real package. Do NOT modify the docs site.
2. Re-run `find-the-gaps analyze --repo ./testdata/fixtures/known-good --docs-url https://<known-good-docs>`.
3. Inspect the report.

**Success Criteria**:
- [ ] Exit code signals findings (non-zero, or zero with a findings summary — policy documented in the implementation).
- [ ] Report contains a finding that names `Frobnicate` and classifies it as missing/undocumented.
- [ ] The finding includes the file path and line number of the new symbol.

**If Blocked**: If the tool reports something unrelated instead, capture its output verbatim and ask the developer before adjusting anything.

---

### Scenario 3: Detect Stale Example

**Context**: Start from the known-good fixture. Modify the signature of a documented function so code and docs disagree.

**Steps**:
1. In the fixture repo, change a documented function's signature (e.g., add a new required parameter).
2. Leave the docs site untouched (it still shows the old signature).
3. Re-run `find-the-gaps analyze ...`.
4. Inspect the report.

**Success Criteria**:
- [ ] Report contains a finding classified as drift / stale example.
- [ ] The finding identifies the function, cites the old signature from docs, and cites the new signature from code.
- [ ] No unrelated findings appear from this single change (spec: only drift-related findings, nothing spurious).

**If Blocked**: If no drift finding is produced, document the exact input and ask the developer — this is a core capability.

---

### Scenario 4: Detect Dead Reference

**Context**: Start from the known-good fixture. Remove a public symbol from code that is still mentioned in docs.

**Steps**:
1. Delete an exported function (and its usages inside the repo) in the fixture.
2. Leave the docs site untouched (it still references the removed symbol).
3. Re-run `find-the-gaps analyze ...`.
4. Inspect the report.

**Success Criteria**:
- [ ] Report contains a finding classified as "removed feature still referenced in docs".
- [ ] The finding names the removed symbol and the doc page that references it.
- [ ] The finding links to the docs page (or provides the URL/path).

**If Blocked**: If the removed symbol is not flagged, stop and ask — this is the reverse direction of Scenario 3 and the tool must handle both.

---

### Scenario 5: Detect Missing Screenshots

**Context**: Known-good fixture + docs site, but a page describes a UI moment with no nearby image.

**Steps**:
1. Run `find-the-gaps analyze --repo ./testdata/fixtures/known-good --docs-url https://<docs>`.
2. Inspect `<projectDir>/screenshots.md`.
3. Re-run with `--skip-screenshot-check`.
4. Inspect the output directory.

**Success Criteria**:
- [ ] First run writes `screenshots.md`; `gaps.md` does NOT contain a `Missing Screenshots` section.
- [ ] `screenshots.md` contains at least one gap for the known UI passage with all four fields populated.
- [ ] Stdout lists `screenshots.md` in the `reports:` block.
- [ ] Second run does NOT write `screenshots.md`.
- [ ] Second run's stdout lists `screenshots.md (skipped)`.

**If Blocked**: If `screenshots.md` renders on the skipped run, the gating is broken. Stop and ask.

---

### Scenario 6: `find-the-gaps doctor`

**Context**: Verify the tool reports external-dependency status accurately.

**Steps**:
1. With `mdfetch` on `$PATH`, run `find-the-gaps doctor`.
2. Temporarily make `mdfetch` unavailable (e.g., rename it in `$PATH` or run with a scrubbed `PATH`).
3. Run `find-the-gaps doctor` again.
4. Restore `mdfetch` and run `find-the-gaps doctor` once more.

**Success Criteria**:
- [ ] In step 1, exit code is `0` and output includes the detected version of `mdfetch`.
- [ ] In step 3, exit code is non-zero and output includes a clear install hint naming `mdfetch` (e.g., `npm install -g @sandgarden/mdfetch`).
- [ ] In step 4, exit code is back to `0`.

**If Blocked**: If version output is incorrect or missing, ask the developer before papering over it.

---

### Scenario 7: First-Run Banner

**Context**: Verify the first-run notification about external dependencies works.

**Steps**:
1. Delete the config dir (`rm -rf ~/.find-the-gaps` or the project-local equivalent).
2. Run any real `find-the-gaps` command (e.g., `find-the-gaps doctor`).
3. Capture the output.
4. Run the same command a second time.
5. Capture the output.
6. Repeat step 2 but set `FIND_THE_GAPS_QUIET=1`.

**Success Criteria**:
- [ ] First invocation prints the external-deps notice naming `mdfetch`.
- [ ] Second invocation does NOT print the notice.
- [ ] Invocation with `FIND_THE_GAPS_QUIET=1` (on a fresh config dir) does NOT print the notice.
- [ ] Config dir exists after the first invocation.

**If Blocked**: If the banner fires every time or never fires, the state tracking is broken — stop and ask.

---

### Scenario 8: Homebrew Install

**Context**: Verify the published Homebrew formula installs cleanly with all dependencies.

**Steps**:
1. On a machine without `find-the-gaps` or `mdfetch` installed (or a fresh VM / clean `brew` test sandbox), run `brew install <tap>/find-the-gaps`.
2. Observe brew's output.
3. Run `find-the-gaps --version`.
4. Run `mdfetch --version`.
5. Run `find-the-gaps doctor`.

**Success Criteria**:
- [ ] `brew install` completes successfully and installs `mdfetch` as a dependency.
- [ ] The caveats block about the external tool is printed.
- [ ] `find-the-gaps --version` returns the installed version.
- [ ] `mdfetch --version` succeeds.
- [ ] `find-the-gaps doctor` exits `0` and reports `mdfetch` present.

**If Blocked**: If brew fails to resolve the `mdfetch` dependency (tap path issue), stop and ask before adjusting the formula live.

---

### Scenario 9: End-to-End on a Real Docs Site

**Context**: Run the full analyze pipeline against a real public docs site and its upstream repo.

**Steps**:
1. Pick a small, real open-source Go project with a public docs site (documented in `testdata/README.md`).
2. Clone the repo at a pinned commit.
3. Run `find-the-gaps analyze --repo <path> --docs-url <url>`.
4. Wait for mdfetch ingestion + analysis to complete.
5. Inspect the report.

**Success Criteria**:
- [ ] `mdfetch` successfully ingests the docs site (markdown files produced in the expected cache location).
- [ ] Analysis completes without panics, timeouts, or Bifrost errors.
- [ ] Report is non-empty and each finding is actionable (names a specific file/symbol/doc page).
- [ ] No finding references a symbol or path that does not exist in the repo.

**If Blocked**: If the docs site changes between runs and breaks the pin, document the new state and ask the developer whether to re-pin the fixture.

---

### Scenario 10: GitHub Action

**Context**: Verify the composite action installs binaries, runs analysis, uploads the artifact, and manages the tracking issue per spec.

**Steps**:
1. Tag a release of this repo (or use the most recent published tag).
2. In a fixture GitHub repository, add a workflow that calls `sandgardenhq/find-the-gaps@<tag>` with a real `docs-url`, `BIFROST_API_KEY` secret, `create-issue: 'true'`.
3. Trigger the workflow via `workflow_dispatch`.
4. Wait for completion. Inspect: artifact, issues tab, run logs.
5. Manually edit the fixture repo to introduce a new exported function (mirrors Scenario 2).
6. Re-trigger the workflow.
7. Inspect the issue (should be edited, not duplicated).
8. Close the issue manually.
9. Re-trigger the workflow.
10. Inspect: a new issue should NOT be created if findings are unchanged from step 7.
11. Re-run with `create-issue: 'false'`. Confirm artifact only.

**Success Criteria**:
- [ ] Step 4: artifact `find-the-gaps-report-<run_id>` is uploaded; issue exists with label `find-the-gaps` and the expected title; run exits `0`.
- [ ] Step 7: same issue number as step 4, body updated to reflect step-5 change.
- [ ] Step 10: no new issue created — closed issues stay dismissed.
- [ ] Step 11: no issue created or modified; only the artifact is produced.

**If Blocked**: If the action fails to download the release binary, capture the URL it tried and ask the developer — the asset-naming convention may have drifted.

---

## Verification Rules

- **Never use mocks or fakes.** All binaries, all network calls, all LLM calls are real.
- Test environments must be fully running copies of real systems (real Bifrost endpoint, real `mdfetch`, real docs site).
- If any success criterion fails, verification fails — partial success is failure.
- If blocked, stop and ask the developer. Do not guess, do not skip, do not mark the task complete.
- All scenarios must pass before a task that touched analysis, installation, or CLI UX is considered done.
